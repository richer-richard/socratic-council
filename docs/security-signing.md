# Code Signing, Notarization, and Auto-Updater Setup

This document explains how to turn on code-signing, notarization, and the
Tauri auto-updater for production releases. The upgrade proposal wave 1.6
registered the updater plugin and set `hardenedRuntime: true` so the wiring
is ready; the remaining steps require your Apple Developer ID and a one-time
generated update-signing keypair.

## 1. Generate the update-signing keypair

Run once, store `~/.tauri/socratic-council.key` somewhere safe (a password
manager, `1Password CLI`, `pass`, etc.). **Do not commit it to the repo.**

```bash
pnpm --filter @socratic-council/desktop tauri signer generate \
  -w ~/.tauri/socratic-council.key
```

Copy the public key that the command prints. Paste it into
`apps/desktop/src-tauri/tauri.conf.json` under `plugins.updater.pubkey`:

```json
"plugins": {
  "updater": {
    "pubkey": "PASTE_PUBLIC_KEY_HERE",
    ...
  }
}
```

Store the private key contents and the passphrase as GitHub Actions secrets:

- `TAURI_SIGNING_PRIVATE_KEY` — the contents of the generated `.key` file
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` — the passphrase you set during generation

## 2. Obtain an Apple Developer ID Application certificate

In Xcode:

1. Xcode → Settings → Accounts → add your Apple Developer account
2. Manage Certificates → + → "Developer ID Application"
3. Open Keychain Access, export the cert as `.p12` (with a password)

Store as GitHub Actions secrets:

- `APPLE_CERTIFICATE` — base64 of the `.p12` file:
  `base64 -i DeveloperIDApplication.p12 | pbcopy`
- `APPLE_CERTIFICATE_PASSWORD` — the password you used for the export
- `APPLE_SIGNING_IDENTITY` — the full cert name, e.g.
  `Developer ID Application: Your Name (TEAMID1234)`
- `APPLE_ID` — your Apple Developer account email
- `APPLE_PASSWORD` — an app-specific password created at
  <https://appleid.apple.com/account/manage> → Security → App-Specific Passwords
- `APPLE_TEAM_ID` — your 10-character team ID

## 3. Add the signed release job

The existing `.github/workflows/release.yml` produces **source** releases.
Add a new file `.github/workflows/desktop-release.yml` that produces signed,
notarized, updater-ready bundles. A template:

```yaml
name: Desktop Release
on:
  push:
    tags: ["v*"]
jobs:
  build-signed:
    runs-on: macos-14
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v3
        with: { version: 9 }
      - uses: actions/setup-node@v4
        with: { node-version: 22, cache: pnpm }
      - uses: dtolnay/rust-toolchain@stable
      - uses: Swatinem/rust-cache@v2
        with: { workspaces: apps/desktop/src-tauri -> target }
      - run: pnpm install --frozen-lockfile
      - name: Build & sign
        working-directory: apps/desktop
        env:
          TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
          TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
          APPLE_CERTIFICATE: ${{ secrets.APPLE_CERTIFICATE }}
          APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERTIFICATE_PASSWORD }}
          APPLE_SIGNING_IDENTITY: ${{ secrets.APPLE_SIGNING_IDENTITY }}
          APPLE_ID: ${{ secrets.APPLE_ID }}
          APPLE_PASSWORD: ${{ secrets.APPLE_PASSWORD }}
          APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
        run: |
          pnpm tauri build \
            --config '{"bundle":{"createUpdaterArtifacts":true}}'
      - name: Publish updater manifest
        env:
          GH_TOKEN: ${{ github.token }}
        run: |
          VERSION=${GITHUB_REF#refs/tags/v}
          gh release upload "v${VERSION}" \
            apps/desktop/src-tauri/target/release/bundle/macos/*.app.tar.gz \
            apps/desktop/src-tauri/target/release/bundle/macos/*.app.tar.gz.sig \
            apps/desktop/src-tauri/target/release/bundle/dmg/*.dmg \
            --clobber
          # latest.json is generated by Tauri and must be uploaded for the
          # updater endpoint in tauri.conf.json to resolve.
```

The Tauri build automatically notarizes when all the `APPLE_*` env vars are
set and `hardenedRuntime: true` is in `tauri.conf.json` (already done).

## 4. Windows Authenticode signing

Add secrets:

- `WINDOWS_CERTIFICATE` — base64 of a `.pfx` file from a trusted CA
- `WINDOWS_CERTIFICATE_PASSWORD` — password for the `.pfx`

Then in a Windows build job:

```yaml
- uses: microsoft/setup-msbuild@v1
- name: Decode Windows cert
  run: |
    echo "${{ secrets.WINDOWS_CERTIFICATE }}" | base64 -d > cert.pfx
- name: Build
  env:
    TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
    TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
  run: pnpm tauri build --config '{"bundle":{"createUpdaterArtifacts":true,"windows":{"certificateThumbprint":"<thumbprint>","digestAlgorithm":"sha256","timestampUrl":"http://timestamp.digicert.com"}}}'
```

## 5. Verify the updater end to end

Once the pubkey is in `tauri.conf.json` and the first signed release has
uploaded `latest.json`, the desktop app will automatically check for updates
at launch (release builds only). To test before a public release:

1. Bump `version` in `apps/desktop/src-tauri/tauri.conf.json`
2. Tag and push (`git tag v1.0.1 && git push --tags`)
3. Install the `v1.0.0` bundle on a clean machine
4. Launch; the updater should surface a toast within a few seconds

If the signature doesn't match the pinned pubkey, the updater rejects the
manifest — this is the defense against a compromised release server pushing
a malicious update.

## What the current repo has configured

- ✅ `tauri-plugin-updater` loaded in release builds (`src/lib.rs`)
- ✅ `plugins.updater.endpoints` points at GitHub Releases
- ✅ `bundle.macOS.hardenedRuntime` is `true`
- ✅ `entitlements.plist` enables sandbox + network.client + files.user-selected.read-write
- ✅ `capabilities/default.json` narrows fs write scope to `$DOWNLOAD/**` and `$DOCUMENT/SocraticCouncil/**`
- ⚠️ `plugins.updater.pubkey` is empty — must be populated before shipping
- ⚠️ `bundle.macOS.signingIdentity` is `null` — CI must provide via env

Until the pubkey and signing identity are set, production builds will not
actually check for updates or be notarized. Source-install via
`install.sh` remains unaffected.
